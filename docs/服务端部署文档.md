# 服务端部署文档

本文档说明在**数据库独立部署**前提下，AI Beauty Muse 后端的完整部署方案，包括：① 数据库的独立部署与建表；② 应用服务的部署方式（直接运行 / systemd / Docker）、环境变量与验证。

---

## 1. 部署架构概览

```
                    ┌─────────────────────────────────────────┐
                    │           应用服务器（本部署文档）          │
                    │  ┌─────────────────────────────────────┐  │
                    │  │  FastAPI 应用 (uvicorn)             │  │
                    │  │  - 认证 / 会员 / 历史报告 / 分析等   │  │
                    │  │  - 上传目录 uploads/ 持久化         │  │
                    │  └──────────────┬──────────────────────┘  │
                    └────────────────┼─────────────────────────┘
                                     │ DATABASE_URL
                                     ▼
                    ┌─────────────────────────────────────────┐
                    │        数据库服务器（独立部署）           │
                    │  MySQL 8+ 或 PostgreSQL 14+              │
                    │  - users / sms_codes / memberships      │
                    │  - usage_quotas / report_history       │
                    │  - chat_sessions / chat_messages        │
                    └─────────────────────────────────────────┘
```

- **数据库**：单独安装与配置，创建库、用户并执行建表 SQL（或由应用 `init_db` 建表）。
- **应用服务**：通过 `DATABASE_URL` 连接上述数据库；可选用直接运行、systemd 或 Docker 部署。

---

## 2. 一、数据库独立部署

### 2.1 何时需要独立部署数据库

| 场景 | 说明 |
|------|------|
| **默认（开发/小规模）** | 使用项目内 SQLite 文件 `./ai_beauty_muse.db`，无需单独部署，应用启动时自动建表。 |
| **生产 / 多实例** | 应用与数据库分离，便于备份、扩容、高可用；多台应用实例共享同一数据库。 |
| **合规 / 运维** | 由 DBA 统一管理数据库、执行建表与迁移，应用仅通过连接串访问。 |

### 2.2 数据库部署流程

1. 安装数据库服务（MySQL 8+ 或 PostgreSQL 14+）。
2. 创建数据库与用户，授权。
3. 执行下方建表 SQL（见 2.5、2.6 节）或由应用 `init_db` 建表。
4. 应用侧配置 `DATABASE_URL` 并安装对应异步驱动（见 3.2、3.3 节）。

### 2.3 支持的数据库与连接格式

| 数据库 | 推荐版本 | Python 异步驱动 | 连接 URL 格式 |
|--------|----------|-----------------|-------------------------------|
| **MySQL** | 8.0+ | `asyncmy` | `mysql+asyncmy://用户:密码@主机:端口/数据库名` |
| **PostgreSQL** | 14+ | `asyncpg` | `postgresql+asyncpg://用户:密码@主机:端口/数据库名` |
| SQLite（默认） | — | `aiosqlite` | `sqlite+aiosqlite:///./ai_beauty_muse.db` |

### 2.4 创建数据库与用户（示例）

**MySQL 8：**

```sql
CREATE DATABASE ai_beauty_muse CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'beauty_user'@'%' IDENTIFIED BY 'YourPassword';
GRANT ALL PRIVILEGES ON ai_beauty_muse.* TO 'beauty_user'@'%';
FLUSH PRIVILEGES;
```

**PostgreSQL 14：**

```sql
CREATE DATABASE ai_beauty_muse ENCODING 'UTF8' LC_COLLATE 'en_US.utf8' LC_CTYPE 'en_US.utf8';
CREATE USER beauty_user WITH PASSWORD 'YourPassword';
GRANT ALL PRIVILEGES ON DATABASE ai_beauty_muse TO beauty_user;
-- \c ai_beauty_muse
-- GRANT ALL ON SCHEMA public TO beauty_user;
```

### 2.5 数据表创建（MySQL 8）

以下 SQL 在 MySQL 8.0+ 下执行，库名请与 `DATABASE_URL` 中一致（如 `ai_beauty_muse`）。执行前：`USE ai_beauty_muse;`

```sql
-- ----------------------------------------------------------------
-- AI Beauty Muse - MySQL 8 建表脚本
-- ----------------------------------------------------------------

CREATE TABLE users (
    id VARCHAR(36) PRIMARY KEY,
    phone VARCHAR(20) UNIQUE NULL,
    device_id VARCHAR(128) UNIQUE NULL,
    nickname VARCHAR(64) NULL,
    avatar_url VARCHAR(500) NULL,
    is_member TINYINT(1) NOT NULL DEFAULT 0,
    member_expires_at DATETIME NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX ix_users_phone (phone),
    INDEX ix_users_device_id (device_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE sms_codes (
    id INT AUTO_INCREMENT PRIMARY KEY,
    phone VARCHAR(20) NOT NULL,
    code VARCHAR(10) NOT NULL,
    purpose VARCHAR(20) NOT NULL DEFAULT 'login',
    expires_at DATETIME NOT NULL,
    used TINYINT(1) NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX ix_sms_codes_phone (phone)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE memberships (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    plan_name VARCHAR(50) NOT NULL DEFAULT 'monthly',
    price DOUBLE NOT NULL,
    started_at DATETIME NOT NULL,
    expires_at DATETIME NOT NULL,
    payment_status VARCHAR(20) NOT NULL DEFAULT 'pending',
    payment_order_id VARCHAR(128) NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX ix_memberships_user_id (user_id),
    CONSTRAINT fk_memberships_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE usage_quotas (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    feature VARCHAR(50) NOT NULL,
    year_month VARCHAR(7) NOT NULL,
    used_count INT NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uq_user_feature_month (user_id, feature, year_month),
    INDEX ix_usage_quotas_user_feature (user_id, feature),
    CONSTRAINT fk_usage_quotas_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE report_history (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    report_type VARCHAR(50) NOT NULL,
    title VARCHAR(200) NOT NULL,
    summary VARCHAR(500) NULL,
    thumbnail_url VARCHAR(500) NULL,
    data LONGTEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX ix_report_history_user_type (user_id, report_type),
    INDEX ix_report_history_created (user_id, created_at),
    CONSTRAINT fk_report_history_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE chat_sessions (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    title VARCHAR(200) NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX ix_chat_sessions_user_id (user_id),
    CONSTRAINT fk_chat_sessions_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE chat_messages (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(36) NOT NULL,
    role VARCHAR(16) NOT NULL,
    content LONGTEXT NOT NULL,
    image_url LONGTEXT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX ix_chat_messages_session_id (session_id),
    CONSTRAINT fk_chat_messages_session FOREIGN KEY (session_id) REFERENCES chat_sessions(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 2.6 数据表创建（PostgreSQL 14）

以下 SQL 在 PostgreSQL 14+ 下执行，需已连接到目标数据库（`\c ai_beauty_muse`）。

```sql
-- ----------------------------------------------------------------
-- AI Beauty Muse - PostgreSQL 14 建表脚本
-- ----------------------------------------------------------------

CREATE TABLE users (
    id VARCHAR(36) PRIMARY KEY,
    phone VARCHAR(20) UNIQUE,
    device_id VARCHAR(128) UNIQUE,
    nickname VARCHAR(64),
    avatar_url VARCHAR(500),
    is_member BOOLEAN NOT NULL DEFAULT FALSE,
    member_expires_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ix_users_phone ON users(phone);
CREATE INDEX ix_users_device_id ON users(device_id);

CREATE TABLE sms_codes (
    id SERIAL PRIMARY KEY,
    phone VARCHAR(20) NOT NULL,
    code VARCHAR(10) NOT NULL,
    purpose VARCHAR(20) NOT NULL DEFAULT 'login',
    expires_at TIMESTAMP NOT NULL,
    used BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ix_sms_codes_phone ON sms_codes(phone);

CREATE TABLE memberships (
    id SERIAL PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    plan_name VARCHAR(50) NOT NULL DEFAULT 'monthly',
    price DOUBLE PRECISION NOT NULL,
    started_at TIMESTAMP NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    payment_status VARCHAR(20) NOT NULL DEFAULT 'pending',
    payment_order_id VARCHAR(128),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ix_memberships_user_id ON memberships(user_id);

CREATE TABLE usage_quotas (
    id SERIAL PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    feature VARCHAR(50) NOT NULL,
    year_month VARCHAR(7) NOT NULL,
    used_count INTEGER NOT NULL DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE (user_id, feature, year_month)
);
CREATE INDEX ix_usage_quotas_user_feature ON usage_quotas(user_id, feature);

CREATE TABLE report_history (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    report_type VARCHAR(50) NOT NULL,
    title VARCHAR(200) NOT NULL,
    summary VARCHAR(500),
    thumbnail_url VARCHAR(500),
    data TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ix_report_history_user_type ON report_history(user_id, report_type);
CREATE INDEX ix_report_history_created ON report_history(user_id, created_at);

CREATE TABLE chat_sessions (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title VARCHAR(200),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ix_chat_sessions_user_id ON chat_sessions(user_id);

CREATE TABLE chat_messages (
    id SERIAL PRIMARY KEY,
    session_id VARCHAR(36) NOT NULL REFERENCES chat_sessions(id) ON DELETE CASCADE,
    role VARCHAR(16) NOT NULL,
    content TEXT NOT NULL,
    image_url TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ix_chat_messages_session_id ON chat_messages(session_id);
```

### 2.7 表结构说明

| 表名 | 说明 |
|------|------|
| **users** | 用户：手机号/设备号、昵称、头像、是否会员及到期时间。 |
| **sms_codes** | 短信验证码：手机号、验证码、用途、过期时间、是否已使用。 |
| **memberships** | 会员订阅记录：用户、方案、价格、起止时间、支付状态、订单号。 |
| **usage_quotas** | 功能配额：用户、功能标识、年月、已用次数。 |
| **report_history** | 历史报告：用户、报告类型、标题、摘要、缩略图、完整 JSON。 |
| **chat_sessions** | 聊天会话：用户、会话标题。 |
| **chat_messages** | 聊天消息：会话、角色、内容、图片 URL。 |

外键：`memberships` / `usage_quotas` / `report_history` / `chat_sessions` → `users.id`（ON DELETE CASCADE）；`chat_messages` → `chat_sessions.id`（ON DELETE CASCADE）。

### 2.8 建表方式选择

| 方式 | 适用场景 |
|------|----------|
| **执行上述 SQL** | DBA 独立建表、严格管控表结构或权限。 |
| **应用 init_db** | 配置好 `DATABASE_URL` 后启动应用，由 `init_db()` 根据 ORM 自动建表（表已存在则跳过）。 |

## 3. 二、应用服务部署

在数据库已独立部署、并完成建表的前提下，按以下方式部署 FastAPI 应用。

### 3.1 环境要求

- **Python**：3.11+
- **网络**：应用服务器能访问数据库（对应端口），若使用 OpenAI/Gemini 需能访问外网或代理。
- **磁盘**：`uploads/` 目录需持久化（用户上传与生成图片）。

### 3.2 环境变量清单（生产必配）

以下为与部署强相关的环境变量，可通过 `.env` 或系统环境传入（`app/config.py` 使用 pydantic-settings 读取）。

| 变量 | 必填 | 说明 | 示例 |
|------|------|------|------|
| **DATABASE_URL** | 是（独立 DB 时） | 数据库连接串 | `mysql+asyncmy://user:pass@host:3306/ai_beauty_muse` |
| **SECRET_KEY** | 是 | JWT 签名密钥，生产务必更换 | 随机长字符串 |
| **HOST** | 否 | 监听地址 | `0.0.0.0` |
| **PORT** | 否 | 监听端口（默认 19000，Docker 常用 8000） | `19000` |
| **CORS_ORIGINS** | 建议 | 允许的前端来源，JSON 数组字符串 | `["https://your-app.com"]` |
| **OPENAI_API_KEY** / **GEMINI_API_KEY** | 按使用 | 使用的 AI 服务 API Key | — |
| **HTTP_PROXY** | 按需 | 出口 HTTP 代理 | `http://127.0.0.1:7890` |
| **UPLOAD_DIR** | 否 | 上传根目录，需持久化 | `uploads` |
| **SMS_PROVIDER** | 生产建议 | 短信服务：`mock` / `aliyun` / `tencent` | `aliyun` |

其他如 `LLM_PROVIDER`、`IMAGE_EDIT_PROVIDER`、`QUOTA_FREE_LIMIT` 等见 `app/config.py`，有默认值。

**.env 示例（数据库独立部署 + 生产）：**

```bash
# 数据库（必配）
DATABASE_URL=mysql+asyncmy://beauty_user:YourPassword@db-host:3306/ai_beauty_muse

# 应用
SECRET_KEY=your-production-secret-key-at-least-32-chars
HOST=0.0.0.0
PORT=19000

# 前端
CORS_ORIGINS=["https://your-app.com","https://www.your-app.com"]

# AI（按实际使用）
LLM_PROVIDER=gemini
GEMINI_API_KEY=your-gemini-key
IMAGE_EDIT_PROVIDER=gemini

# 短信（生产建议改为 aliyun/tencent）
SMS_PROVIDER=aliyun
```

### 3.3 依赖与数据库驱动

基础依赖见 `requirements.txt`。**数据库独立部署时**需安装对应异步驱动：

```bash
# MySQL
pip install -r requirements.txt
pip install asyncmy

# PostgreSQL
pip install -r requirements.txt
pip install asyncpg
```

### 3.4 部署方式一：直接运行（uvicorn）

适用于单机、测试或简单生产。

```bash
# 进入项目目录，建议使用虚拟环境
cd /path/to/ai-beauty-muse-backend
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

pip install -r requirements.txt
pip install asyncmy   # 或 asyncpg

# 确保 .env 或环境变量已配置 DATABASE_URL、SECRET_KEY 等
uvicorn app.main:app --host 0.0.0.0 --port 19000
```

如需指定端口：`uvicorn app.main:app --host 0.0.0.0 --port 8000`。应用启动时会执行 `init_db()`，若库中尚无表会自动创建（使用 SQLite 时默认即如此；使用 MySQL/PostgreSQL 时需已配置 `DATABASE_URL`）。

### 3.5 部署方式二：systemd 守护进程

适用于 Linux 服务器，开机自启、崩溃重启。

**1）创建 systemd 单元文件**（如 `/etc/systemd/system/ai-beauty-muse.service`）：

```ini
[Unit]
Description=AI Beauty Muse Backend
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/path/to/ai-beauty-muse-backend
EnvironmentFile=/path/to/ai-beauty-muse-backend/.env
ExecStart=/path/to/ai-beauty-muse-backend/.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 19000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

**2）启用并启动：**

```bash
sudo systemctl daemon-reload
sudo systemctl enable ai-beauty-muse
sudo systemctl start ai-beauty-muse
sudo systemctl status ai-beauty-muse
```

**3）日志：** `journalctl -u ai-beauty-muse -f`

请将 `WorkingDirectory`、`EnvironmentFile`、`ExecStart` 中的路径和端口改为实际值；若数据库与应用同机，确保数据库服务已配置为 `After=network.target` 或 `After=mysql.service` 等。

### 3.6 部署方式三：Docker

镜像内不包含数据库，仅运行应用，通过 `DATABASE_URL` 连接外部数据库。

**1）构建镜像：**

当前仓库根目录的 `Dockerfile` 暴露端口为 8000，如需与 config 默认 19000 一致，可在运行阶段通过环境变量覆盖（见下）。

```bash
cd /path/to/ai-beauty-muse-backend
docker build -t ai-beauty-muse-backend:latest .
```

**2）运行容器（连接独立数据库）：**

```bash
docker run -d \
  --name ai-beauty-muse \
  -p 19000:19000 \
  -e DATABASE_URL="mysql+asyncmy://beauty_user:YourPassword@host.docker.internal:3306/ai_beauty_muse" \
  -e SECRET_KEY="your-production-secret-key" \
  -e GEMINI_API_KEY="your-gemini-key" \
  -e PORT=19000 \
  -v /path/on/host/uploads:/app/uploads \
  ai-beauty-muse-backend:latest \
  uvicorn app.main:app --host 0.0.0.0 --port 19000
```

说明：

- **DATABASE_URL**：若数据库在宿主机，可用 `host.docker.internal`（Mac/Windows）或宿主机内网 IP；若数据库在另一台机器，填该机 IP 或域名。
- **uploads**：`-v` 将宿主机目录挂载到容器内 `/app/uploads`，保证上传文件持久化。
- 镜像默认未安装 `asyncmy`/`asyncpg`，若使用 MySQL/PostgreSQL，需在 **Dockerfile 中增加** 对应驱动安装，例如：
  ```dockerfile
  RUN pip install --no-cache-dir -r requirements.txt && pip install asyncmy
  ```
  或单独 `requirements-mysql.txt` 包含 `asyncmy`，再在 Dockerfile 中 `RUN pip install -r requirements-mysql.txt`。

**3）docker-compose 示例（应用 + 仅数据库同机）：**

可在同一 compose 中只启动应用，数据库由外部提供；或 compose 内同时定义 MySQL/PostgreSQL 与 app 服务，由 app 通过服务名连接。示例（仅应用，数据库假设已存在）：

```yaml
# docker-compose.yml 片段
services:
  app:
    build: .
    ports:
      - "19000:19000"
    env_file: .env
    environment:
      - DATABASE_URL=mysql+asyncmy://beauty_user:pass@db:3306/ai_beauty_muse
      - PORT=19000
    volumes:
      - ./uploads:/app/uploads
    command: uvicorn app.main:app --host 0.0.0.0 --port 19000
```

若数据库也在 compose 中，增加 `depends_on: - db` 并保证 `db` 服务名与 `DATABASE_URL` 中主机名一致。

### 3.7 健康检查与验证

- **健康接口**：`GET /health`，返回 200 表示进程与基础依赖正常（不检测数据库连接）。可用于负载均衡或容器 HEALTHCHECK。
- **数据库与表**：部署后调用需登录的接口（如登录、历史报告列表），能正常返回即说明数据库连接与表结构正常。
- **上传目录**：确认 `uploads/`（或 `UPLOAD_DIR`）存在且进程有写权限；发色实验/发型编辑等接口会写入该目录。

## 4. 附录

### 4.1 从 SQLite 迁移到独立库

若此前使用 SQLite 文件：

1. 在新库中按 2.5 或 2.6 节执行建表 SQL。
2. 使用导出/导入工具或脚本将 SQLite 数据迁移到新库（注意主键、时间、布尔等类型兼容）。
3. 配置 `DATABASE_URL` 并安装对应驱动，重启应用指向新库。

- 应用配置项完整定义：`app/config.py`
- 数据库 ORM 与建表逻辑：`app/models/database.py`，`init_db()` 在 `app/main.py` 启动时调用
