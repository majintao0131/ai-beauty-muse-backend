# 第一部分 产品需求概述

## 1.1 产品定位与功能模块

AI Beauty Muse 是一款**智能美学与形象顾问**产品，基于 AI 提供：

| 模块 | 产品能力 | 典型场景 |
|------|----------|----------|
| **用户与会员** | 手机号登录、会员订阅、功能使用配额（免费/会员次数） | 注册登录、会员购买、次数展示与限制 |
| **面部分析** | 脸型识别、五官与比例分析、面相解读、运势美学（当日幸运色与妆容） | 上传照片获得全面面相报告与今日美学建议 |
| **色彩诊断** | 个人色彩类型、适合/避免色、妆发色推荐 | 根据照片判定四季型并给出用色建议 |
| **身材风格** | 身材类型、扬长避短、廓形与穿搭建议 | 输入三围获得体型分析与穿搭建议 |
| **发型发色** | 脸型+肤色分析 → 3 款发型+3 款发色推荐；AI 换发/换色效果图 | 选发型发色并生成效果图，可生成理发师沟通卡 |
| **AI 实验室** | 发色实验（仅改发色）、自由指令发型/发色编辑 | 一键试发色、自由文案改发 |
| **命理运势** | 八字、喜用神、今日运势与幸运色、场合建议 | 输入生日获得命理解读与今日开运建议 |
| **每日能量** | 当日干支、五行能量、幸运色、穿搭与妆容要点 | 每日打开获取当日能量与穿搭指引 |
| **历史报告** | 所有 AI 结果自动或主动落地为报告，列表/详情/删除 | 「我的报告」查看、筛选、删除与详情展示 |
| **全面落地建议** | 综合脸型、色彩、身材、命理、当日能量生成 2～4 句总述 + 分板块可执行建议 | 多维度分析后一键生成今日/近期行动清单 |
| **清理历史** | 单条删除（APP）、按类型/用户/时间批量删除（后台/数据库） | 用户自主清理、运营/合规批量清理 |

## 1.2 用户认证与会员

- **登录方式**：手机号 + 短信验证码；首次自动注册。可选设备号登录（兼容旧版）。
- **鉴权**：JWT Bearer Token，长期有效（如 30 天），支持刷新。
- **会员**：月费订阅（如 19.9 元/月），订阅后享受更高功能次数上限。
- **配额**：按功能（如 face_style、face_analysis、destiny_color）按月统计，免费用户与会员每月次数不同，超限后需等待下月或开通会员。

## 1.3 面部分析与美学

- **输入**：一张面部照片（可选生日、是否含运势美学）。
- **输出**：脸型与骨相、五官分析（标签+评分+简析）、面部比例、面相解读（事业/财运/感情/健康/性格及今日能量影响）、发型与妆容建议；若开启运势美学，则含当日幸运色、2 套妆容方案、配饰建议及可选 AI 参考图。
- **需求要点**：支持生日个性化幸运色；可关闭运势美学以缩短耗时。

## 1.4 发型发色与 AI 实验室

- **推荐流程**：上传正面照 → 获得 3 款发型 + 3 款发色推荐及整体建议。
- **效果图**：用户选择某条推荐（或自定义文案）后，上传原图 + 指令，生成仅改发型/发色的新图；可多次生成（如 3 张）并可选生成理发师沟通卡。
- **发色实验**：仅改发色，不改发型/脸型；用户选择发色名称即可生成。
- **报告落地**：完整流程（推荐 + 多张效果图 + 可选沟通卡）可由 APP 聚合为一条「AI 发型」报告保存到历史，避免列表零散。

## 1.5 命理与每日能量

- **命理运势**：输入生日（可选时辰、今日场合），获得八字、日主、喜用神、性格与命理总览、今日运势总评与评分、分项运势（事业/感情/财运/健康）、今日幸运色与穿搭建议。
- **每日能量**：可选场合与生日，获得当日干支、五行能量、幸运色、穿搭与妆容要点、能量提示及场合特别建议。

## 1.6 历史报告与报告落地

- **自动保存**：面相分析、发型推荐、发型编辑、发色实验、色彩诊断、身材分析、命理运势、每日能量、理发师沟通卡、全面落地建议等接口成功调用后，若用户已登录，结果会自动写入「我的报告」。
- **主动创建**：APP 可将多次调用结果聚合为一条报告（如 face_style：推荐 + 3 张效果图 + 沟通卡），调用「主动创建报告」接口落地；对 face_style 有固定 data 结构校验。
- **全面落地建议**：传入多模块数据（至少一项），由 AI 生成总述 + 3～5 个分板块可执行建议，并可自动写入历史报告。

## 1.7 清理历史报告（产品侧）

- **用户侧**：在「我的报告」中每条报告提供删除入口（如垃圾桶图标），单条删除；当前无「清空全部」或按类型批量删除，若需要可由 APP 后续扩展。
- **运营/后台**：支持在数据库层按用户、报告类型、时间范围批量删除或去重（见 2.12 节）。

# 第二部分 技术接口文档

## 2.1 通用约定

- **Base URL**：开发/测试如 `http://<host>:19000`，生产以实际部署为准。接口前缀：`/api/v1`。
- **鉴权**：需鉴权的接口在请求头中携带 `Authorization: Bearer <access_token>`。
- **错误响应**：通常为 JSON `{"detail": "错误说明"}`；4xx 为客户端错误，5xx 为服务端错误。
- **图片上传**：支持的格式为 JPEG/PNG/WebP，大小上限 10 MB。返回的图片 URL 多为相对路径（如 `/uploads/edited/xxx.png`），前端需拼接 Base URL 后展示。

## 2.2 用户认证与会员

### 接口列表

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| POST | `/api/v1/auth/sms/send` | 发送短信验证码 | 否 |
| POST | `/api/v1/auth/sms/login` | 手机号+验证码登录/注册 | 否 |
| POST | `/api/v1/auth/refresh` | 刷新 Token | 是 |
| GET  | `/api/v1/auth/me` | 获取个人信息与配额 | 是 |
| POST | `/api/v1/membership/subscribe` | 订阅会员 | 是 |
| GET  | `/api/v1/membership/status` | 会员状态 | 是 |
| GET  | `/api/v1/membership/quota` | 使用次数查询 | 是 |

### 发送验证码 `POST /api/v1/auth/sms/send`

- **请求体**：`{"phone": "13812345678"}`，手机号需符合 `^1[3-9]\d{9}$`。
- **成功**：`{"success": true, "message": "验证码已发送", "expires_in": 300}`。
- **错误**：422 手机号格式错误；429 发送过于频繁。

### 手机号登录/注册 `POST /api/v1/auth/sms/login`

- **请求体**：`{"phone": "13812345678", "code": "123456", "nickname": "小美"}`（nickname 可选）。
- **成功**：`{"access_token": "...", "token_type": "bearer", "user_id": "...", "phone": "138****5678", "nickname": "小美", "is_member": false}`。
- **错误**：400 验证码错误或过期。

### 获取个人信息 `GET /api/v1/auth/me`

- **Header**：`Authorization: Bearer <token>`。
- **成功**：返回 user_id、phone（脱敏）、nickname、is_member、member_expires_at、各功能配额（used/limit/remaining）等。

### 配额规则

| 用户类型 | face_style / face_analysis / destiny_color |
|----------|--------------------------------------------|
| 免费用户 | 3 次/月（可配置） |
| 会员用户 | 10 次/月（可配置） |

每月按自然月重置；会员到期后按免费额度计。

## 2.3 面部分析

### 接口信息

| 项目 | 说明 |
|------|------|
| 方法 | `POST` |
| 路径 | `/api/v1/analysis/face` |
| Content-Type | `multipart/form-data` |
| 鉴权 | 是 |
| 耗时 | 约 5–30 秒（含运势美学与参考图时更长） |

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| image | File | 是 | 面部照片，JPEG/PNG/WebP，≤10MB |
| birth_year | int | 否 | 出生年（1900–2100），用于个性化幸运色 |
| birth_month | int | 否 | 出生月 |
| birth_day | int | 否 | 出生日 |
| include_fortune | bool | 否 | 是否包含运势美学板块，默认 true |

### 响应要点

- 脸型：face_shape、face_shape_cn；骨相：forehead、cheekbones、jawline、chin。
- 五官：five_features（眉/眼/鼻/口/耳，每项 tag、score、文字）。
- 比例：face_proportions（三庭五眼、对称性）。
- 面相解读：face_reading（事业/财运/感情/健康/性格及今日能量）。
- 运势美学（include_fortune 为 true 时）：fortune_beauty（date、daily_stem_branch、lucky_colors、makeup_looks、accessories、look_image_url 等）。
- 建议：hairstyle_recommendations、makeup_tips。
- 上传图可返回 input_image_url 供列表缩略图使用。

## 2.4 色彩诊断与身材分析

### 色彩诊断 `POST /api/v1/analysis/color`

- **请求体**：JSON，`{"image_url": "可访问的脸部/皮肤图片 URL"}`。
- **响应**：season_type、season_type_cn、skin_undertone、best_colors、avoid_colors、neutral_colors、makeup_colors、hair_colors、analysis。颜色项含 name、hex、element 等。
- **鉴权**：以实际实现为准；登录用户结果可自动写入报告 color_diagnosis。

### 身材分析 `POST /api/v1/analysis/body`

- **请求体**：JSON，`{"height": 165, "bust": 86, "waist": 70, "hip": 92}`（单位 cm，范围见接口约束）。
- **响应**：body_type、body_type_cn、body_type_description、proportions、strengths、areas_to_enhance、recommended_silhouettes、styles_to_avoid、accessories。
- **鉴权**：以实际实现为准；登录用户结果可自动写入报告 body_analysis。

## 2.5 发型发色推荐与图片编辑

### 发型发色推荐 `POST /api/v1/analysis/face-style`

- **Content-Type**：multipart/form-data。
- **参数**：image（File，必填），JPEG/PNG/WebP，≤10MB。
- **鉴权**：是（并占用 face_style 配额）。
- **响应**：face_shape、face_shape_cn、face_analysis、skin_tone、hairstyle_recommendations（3 条，含 name、description、length、styling_tips）、hair_color_recommendations（3 条，含 color_name、color_hex、reason）、overall_advice；可选 input_image_url。

### AI 换发照片（文本指令）`POST /api/v1/analysis/face-edit`

- **Content-Type**：multipart/form-data。
- **参数**：image（File，必填），instructions（string，必填，如「改成齐耳短发，蜜茶棕色」），provider（可选，gemini / gpt-image-1）。
- **鉴权**：可选；登录用户结果可写入报告 face_edit。
- **响应**：image_url、image_path、modification_applied、provider。
- **说明**：用于理髮師溝通卡「文本描述」模式：用户上传人像 + 填写发型/发色文字描述 → 生成效果图 → 再调沟通卡接口。

### 根据参考图换发 `POST /api/v1/analysis/face-edit-by-reference`

- **Content-Type**：multipart/form-data。
- **参数**：image（File，必填，用户人像照片），reference_image（File，必填，发型参考图）。
- **鉴权**：可选；登录用户结果可写入报告 face_edit。
- **响应**：与 face-edit 相同结构（image_url、image_path、modification_applied、provider）。image_url 可为相对路径，客户端拼接 baseUrl 使用。
- **说明**：用于理髮師溝通卡「发型参考图」模式：用户上传人像 + 上传一张发型参考图 → 后端将参考图上的发型迁移到人像上生成效果图 → 再调沟通卡接口。当前实现使用 Gemini；效果图可来自本接口或 face-edit 任一，再调用 stylist-card 生成沟通卡。

## 2.6 AI 实验室：发色实验与图片编辑

### 发色实验 `POST /api/v1/analysis/hair-color-experiment`

- **Content-Type**：multipart/form-data。
- **参数**：image（File，必填），hair_color（string，必填，如蜜茶棕、栗棕色、雾霾蓝），provider（可选）。
- **鉴权**：可选；登录用户结果可写入报告 hair_color_experiment。
- **响应**：与 face-edit 相同结构（image_url、modification_applied、provider 等）。
- **说明**：仅改发色，不改发型/脸型/背景。

### 图片规范（发色实验与 face-edit 通用）

- 格式：image/jpeg、image/png、image/webp；大小 ≤10MB。建议正面人像、光线均匀。
- 返回 image_url 为相对路径，前端需拼接 Base URL 后展示或下载。

## 2.9 理发师沟通卡

理髮師溝通卡支持兩種輸入方式：

1. **文本描述**：用户上传人像照 + 填写发型/发色文字描述 → 调用「人像换发」`POST /api/v1/analysis/face-edit` 生成效果图 → 再调用「沟通卡」生成参数卡。
2. **发型参考图**：用户上传人像照 + 上传一张发型参考图 → 调用「根据参考图换发」`POST /api/v1/analysis/face-edit-by-reference` 生成效果图 → 再调用「沟通卡」生成参数卡。

| 输入方式   | 步骤 1 | 步骤 2 |
|------------|--------|--------|
| 文本描述   | `POST /api/v1/analysis/face-edit`（image + instructions） | `POST /api/v1/hairstyle/stylist-card`（effect_image_url + 描述等） |
| 发型参考图 | `POST /api/v1/analysis/face-edit-by-reference`（image + reference_image） | `POST /api/v1/hairstyle/stylist-card`（effect_image_url + 固定描述） |

### 生成理髮師溝通卡 `POST /api/v1/hairstyle/stylist-card`

根据效果图 URL 与发型描述等参数，生成专业技术参数卡（修剪指导、染发参数、造型维护等）。效果图可来自上述 face-edit 或 face-edit-by-reference 任一接口。

| 项目 | 说明 |
|------|------|
| 方法 | `POST` |
| 路径 | `/api/v1/hairstyle/stylist-card` |
| Content-Type | `application/json` |
| 鉴权 | 以实际实现为准；登录用户结果可写入报告 stylist_card |

### 请求体（核心字段）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| hairstyle_name | string | 是 | 选定的发型名称 |
| hairstyle_description | string | 是 | 发型描述（为何适合该脸型） |
| hairstyle_length | string | 是 | short / medium / long |
| styling_tips | string | 否 | 造型建议 |
| color_name | string | 否 | 发色名称 |
| color_hex | string | 否 | 发色十六进制，如 #9CA3AF |
| face_shape | string | 否 | 脸型英文 |
| face_shape_cn | string | 否 | 脸型中文 |
| face_analysis | string | 否 | 面部分析摘要 |
| skin_tone | string | 否 | 肤色描述 |
| effect_image_url | string | **是** | 效果图完整 URL（客户端已拼接 baseUrl），来自 face-edit 或 face-edit-by-reference 的 image_url |

当输入方式为「发型参考图」时，客户端可传 hairstyle_name: "参考图发型"、hairstyle_description: "根据上传的发型参考图生成"。以上其余字段多来自 face-style 与 face-edit 结果。

### 响应要点

- **card_image_url**（可选）：若后端生成整张沟通卡图片可返回，客户端目前以文字区块为主展示。
- **cutting_guide**：outline、technique、weight_balance、key_points（数组）。
- **color_formula**：formula_code、formula_name、brand_reference、brand_alt、bleach_required、bleach_level、processing_time、energy_note。
- **styling_guide**：daily_routine、products（数组）、tools（数组）、maintenance_cycle。
- **detail_notes**：数组，每项 area、note（如刘海军区备注）。

## 2.10 历史报告

### 接口列表

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| GET | `/api/v1/history/reports` | 报告列表（分页、按类型筛选） | 是 |

### 列表 `GET /api/v1/history/reports`

- **查询参数**：report_type（可选）、page（默认 1）、page_size（默认 20，最大 100）。
- **响应**：reports（id、report_type、title、summary、thumbnail_url、created_at）、total、page、page_size。

### 详情 `GET /api/v1/history/reports/{report_id}`

- **响应**：id、report_type、title、summary、thumbnail_url、data（完整报告 JSON）、created_at。data 结构与对应功能接口响应一致，可按 report_type 渲染不同详情页。

### 删除 `DELETE /api/v1/history/reports/{report_id}`

- **响应**：200 时 `{"success": true, "message": "报告已删除"}`；404 表示不存在或无权限。

### 报告类型 `GET /api/v1/history/types`

- **响应**：types 数组，每项含 key（如 face_analysis）、label（如「面相分析」）。用于筛选与展示。

## 2.11 报告落地与全面落地建议

### 主动创建报告 `POST /api/v1/history/reports`

- **鉴权**：必填。
- **请求体**：report_type（必填，见上表）、title（必填，≤200 字）、summary（可选）、thumbnail_url（可选）、data（必填，JSON 对象）。当 report_type 为 face_style 时，data 须符合下方 face_style 建模。
- **成功**：`{"id": "报告UUID", "success": true}`。
- **错误**：400 report_type 不合法或 data 结构不符合建模；401 未登录。

### face_style 的 data 结构（建模）

- **顶层**：face_style（object，可选）、combinations（array）、stylist_card（object，可选）。
- **face_style**：face_shape、face_shape_cn、face_analysis、skin_tone、overall_advice 等。
- **combinations[]**：id、hairstyle（name、length、description、styling_tips）、hair_color（color_name、color_hex、reason）、edit_result（**必须含 image_url**，与 face-edit 响应一致）。
- 若某组合要展示效果图，则 edit_result.image_url 必填，否则后端校验报 400。

### 全面落地建议 `POST /api/v1/analysis/landing-suggestion`

- **Content-Type**：application/json。
- **鉴权**：可选；登录用户结果可写入报告 landing_suggestion。
- **请求体**：至少传入以下一项（结构与对应分析接口响应或报告 data 一致）：face_analysis、color_diagnosis、body_analysis、destiny_fortune、daily_energy。
- **响应**：summary（总述）、sections（数组，每项 title、content）、report_id（若已写入历史则返回 UUID，否则 null）。
- **错误**：400 未传任何一项；500 生成失败。

## 2.12 清理历史报告（后台/数据库）

除 APP 内单条删除（调用 `DELETE /api/v1/history/reports/{report_id}`）外，可在数据库层批量清理。表名：`report_history`，主要字段：id、user_id、report_type、title、summary、thumbnail_url、data、created_at。

### 1. 只删某用户的某类报告

```sql
DELETE FROM report_history
WHERE user_id = 'YOUR_USER_ID'
  AND report_type = 'face_analysis';
```

### 2. 删除某类型全库

```sql
DELETE FROM report_history
WHERE report_type = 'face_analysis';
```

### 3. 删除重复项（同用户、同标题保留最新一条，以 PostgreSQL 为例）

```sql
DELETE FROM report_history r
USING report_history r2
WHERE r.report_type = 'face_analysis'
  AND r.user_id = r2.user_id
  AND r.title = r2.title
  AND r.created_at < r2.created_at;
```

### 4. 只删超过 N 天的旧报告

```sql
-- 示例：删除 90 天前的面相分析报告
DELETE FROM report_history
WHERE report_type = 'face_analysis'
  AND created_at < NOW() - INTERVAL '90 days';
```

（MySQL 中可将 `INTERVAL '90 days'` 改为 `INTERVAL 90 DAY`。）

### 注意

- 执行 DELETE 前建议先用 SELECT 确认范围。
- 生产环境操作前请备份数据库或在测试环境验证。
